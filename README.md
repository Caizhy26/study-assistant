# 学习助手 v2.1

一个面向大学生复习规划场景的学习助手项目。这个版本已经从“纯前端直连模型”升级为“前端页面 + 后端代理接口”的结构：前端负责交互、日程展示和本地状态管理，后端负责安全地连接第三方 AI 服务，并通过 `/api/health`、`/api/chat`、`/api/vision` 三个接口向前端提供能力。

这个 README 适合用来向同学、老师或后续维护者说明：这个版本做了什么、为什么要这样拆、怎么本地运行、怎么部署、出现问题时应该先查哪里。

---

## 1. 版本定位

这个版本的核心目标不是只“把页面做出来”，而是把项目推进到**可以真实部署、可以安全连上模型、可以继续迭代**的状态。

相比早期版本，这一版有几个重要变化：

- 不再由浏览器直接携带 API Key 请求第三方模型。
- 前端统一请求自己的 `/api/*` 接口。
- 新增后端代理层，负责读取环境变量并调用上游模型服务。
- 增加 `/api/health` 健康检查，用来判断当前是否在线可用。
- 保留离线模式：当后端不可用、模型报错或密钥失效时，前端仍能继续完成基础任务生成和信息整理。

一句话概括：

> 这个版本已经不是纯前端演示项目，而是具备“前端交互 + 后端代理 + 在线/离线双模式”的完整应用雏形。

## 2. 项目解决什么问题

这个项目的目标是帮助大学生把“模糊的学习想法”转成“可执行的学习任务”。

例如，用户原本只会说：

- 这周想复习数学分析和线性代数
- 我晚上有空，但不知道怎么安排
- 这章我不会，帮我排进日程里
- 我上传了一张课表图，你帮我识别一下

系统会把这些自然语言输入，转成结构化信息：

- 课程列表
- 用户画像（基础、目标、教材、考试时间等）
- 可用时段
- 当前卡点
- 可写入日程的具体任务

最终让用户看到的不是“泛泛建议”，而是已经进入系统的真实任务和计划。

## 3. 这个版本的主要能力

### 3.1 AI 对话式收集信息

前端提供对话入口，用户可以像和学习规划师聊天一样描述自己的情况。系统会尝试从对话中提取：

- 学习科目
- 当前基础
- 空闲时间
- 具体薄弱点
- 希望生成计划的意图

### 3.2 自动生成学习任务

当用户表达明确排期意图，或者对话达到一定信息量后，系统会生成具体任务，并写入任务列表或日程相关状态中。

### 3.3 课表/图片识别

项目支持上传图片附件。图片会先经过视觉识别接口，提取课表或课程相关信息，再并入用户画像。

### 3.4 在线模式 + 离线模式

- **在线模式**：后端代理可用，且文本/视觉模型配置正确时，调用真实模型。
- **离线模式**：当后端不可用或上游模型报错时，前端会切换到内置规则引擎，不至于整个应用失效。

### 3.5 学习场景页面

项目中已经包含多个学习管理场景页面，例如：

- 首页 / Home
- 任务录入与 AI 对话 / Intake
- 日程规划 / Plan
- 即刻开始 / Now
- 任务列表 / Tasks
- 计时器 / Timer
- 状态管理 / Energy / Review / Report

这些页面共同构成一个轻量学习助手的完整交互闭环。

## 4. 技术结构总览

### 4.1 当前项目属于什么架构

这个项目目前是：

> **逻辑上的前后端分层，代码上的单仓库项目。**

也就是说：

- 前端和后端已经承担不同职责。
- 但它们仍然放在同一个仓库里，方便开发和交付。

### 4.2 前端负责什么

前端代码主要在 `src/` 目录下，负责：

- 页面展示
- 用户输入
- 聊天交互
- 本地状态管理
- 附件上传
- 调用 `/api/*`
- 在线失败时切换离线模式

前端的关键目录和文件：

```text
src/
  App.jsx
  pages/
  components/
  hooks/
  services/aiService.js
  utils/
  styles/
```

### 4.3 后端负责什么

后端代码主要在：

```text
server/index.mjs
```

后端负责：

- 读取 `.env` / `.env.local` 中的环境变量
- 保存敏感配置（如 API Key）
- 暴露本地开发所需的 `/api/*` 接口
- 代理请求到上游模型服务
- 返回统一的 JSON 结果给前端

当前后端提供三个接口：

- `GET /api/health`：健康检查
- `POST /api/chat`：文本模型代理
- `POST /api/vision`：视觉模型代理

## 5. 为什么要做前后端分层

这个版本最重要的工程改进，就是把“模型调用”和“页面交互”拆开。

### 5.1 解决 API Key 安全问题

早期纯前端做法会把 Key 暴露在浏览器里，这会带来：

- 安全风险
- 部署后不稳定
- 不适合正式环境

现在 Key 只保存在服务端环境变量里，浏览器侧不会直接看到它。

### 5.2 解决部署后无法真实连上模型的问题

纯静态页面部署后，常见问题是：

- 页面能打开，但 AI 不能用
- 上游接口鉴权失败时不容易定位
- 文本和视觉接口配置分散

现在统一通过 `/api/chat` 和 `/api/vision` 访问，更容易部署和排查问题。

### 5.3 方便后续扩展

有了后端代理层，后面更容易加：

- 用户登录
- 云端存储
- 调用日志
- 限流
- 多模型切换
- 配额控制
- 审计与安全处理

### 5.4 降低前端耦合

前端不需要知道 DeepSeek 或其他模型接口的全部细节，只需要知道：

- 请求 `/api/chat`
- 请求 `/api/vision`
- 请求 `/api/health`

模型更换、鉴权方式变化、服务端限流等，都可以优先在后端处理。

## 6. 数据流怎么走

### 6.1 文本对话链路

用户在前端输入消息后，数据会按这个路径流动：

```text
用户输入
  ↓
前端页面（IntakePage）
  ↓
services/aiService.js
  ↓
POST /api/chat
  ↓
server/index.mjs
  ↓
第三方文本模型接口
  ↓
返回 JSON
  ↓
前端解析 reply / extractedData / newTasks
  ↓
更新课程、任务、issues、聊天记录
```

### 6.2 图片识别链路

```text
用户上传图片
  ↓
前端读取为 data URL / base64
  ↓
POST /api/vision
  ↓
server/index.mjs
  ↓
第三方视觉模型接口
  ↓
返回识别结果
  ↓
前端合并到课程/课表信息中
```

### 6.3 健康检查链路

```text
页面加载
  ↓
GET /api/health
  ↓
判断后端是否可用、文本/视觉接口是否配置
  ↓
决定是走在线模式还是离线模式
```

## 8. 关键文件说明

### `src/services/aiService.js`

这是前端里最关键的服务层文件之一，负责：

- 构造 `/api/chat` 和 `/api/vision` 请求
- 定义 AI 系统提示词
- 从模型回复中稳健提取 JSON
- 在线失败时走离线 fallback
- 标准化课程名、任务字段、识别结果

你可以把它理解为：

> 前端与 AI 能力之间的“中间层”。

### `server/index.mjs`

这是本地开发时的后端入口，负责：

- 启动一个 Node HTTP 服务
- 暴露 `/api/health`、`/api/chat`、`/api/vision`
- 从环境变量读取 `DEEPSEEK_API_KEY`、`VISION_API_KEY` 等配置
- 把请求代理到第三方模型接口

### `src/pages/IntakePage.jsx`

这是当前最核心的业务页面之一，承担了：

- 对话输入
- 附件处理
- AI 调用入口
- 在线/离线切换
- 提取数据并写入本地状态

这个页面功能很多，也是后续最适合继续拆分优化的地方。

## 9. 环境变量说明

项目提供了 `.env.example` 作为模板。实际运行时需要复制成 `.env`。

推荐的最小配置如下：

```env
VITE_API_BASE_URL=
DEEPSEEK_API_KEY=你的DeepSeekKey
TEXT_API_URL=https://api.deepseek.com/chat/completions
TEXT_MODEL=deepseek-chat
VISION_API_KEY=
VISION_API_URL=https://api.scnet.cn/api/llm/v1/chat/completions
VISION_MODEL=MiniMax-M2.5
PORT=8787
CORS_ORIGIN=*
```

### 字段说明

- `VITE_API_BASE_URL`：前端请求 API 的基础地址。前后端同域时通常留空。
- `DEEPSEEK_API_KEY`：文本模型 Key。
- `TEXT_API_URL`：文本模型接口地址。
- `TEXT_MODEL`：文本模型名称。
- `VISION_API_KEY`：视觉模型 Key，可选。
- `VISION_API_URL`：视觉模型接口地址。
- `VISION_MODEL`：视觉模型名称。
- `PORT`：本地后端服务端口。
- `CORS_ORIGIN`：允许的跨域来源。

## 10. 本地运行方式

### 10.1 环境要求

建议使用：

- Node.js 18+
- npm 9+

### 10.2 安装依赖

```bash
npm install
```

### 10.3 创建环境变量文件

把 `.env.example` 复制为 `.env`，然后补全 `DEEPSEEK_API_KEY`。

Windows 用户如果不想用命令，可以直接在 VS Code 中复制 `.env.example`，重命名为 `.env`。

### 10.4 启动后端

```bash
npm run dev:backend
```

默认监听：

```text
http://localhost:8787
```

### 10.5 启动前端

另开一个终端：

```bash
npm run dev:frontend
```

前端开发服务器默认运行在 Vite 提供的本地地址上。

### 10.6 验证后端是否正常

浏览器打开：

```text
http://localhost:8787/api/health
```

正常情况下会返回类似：

```json
{
  "ok": true,
  "mode": "proxy",
  "chatConfigured": true,
  "visionConfigured": false,
  "textApiUrl": "https://api.deepseek.com/chat/completions",
  "visionApiUrl": "https://api.scnet.cn/api/llm/v1/chat/completions"
}
```

如果 `chatConfigured` 为 `false`，通常说明 `DEEPSEEK_API_KEY` 没填对，或者 `.env` 没有被读取到。

## 12. 部署说明

### 12.1 本地开发和线上部署的区别

本地开发时：

- 前端由 Vite 提供页面
- 后端由 `server/index.mjs` 提供 `/api/*`
- `vite.config.js` 会把 `/api` 请求转发给本地后端

线上部署时：

- 前端会构建为静态文件
- 后端需要继续存在，不能只部署页面
- 如果部署到 Vercel，则通常会把 `/api/*` 改造成 Serverless Functions

### 12.2 为什么不能只部署前端静态页面

因为这个项目现在的在线能力依赖后端代理。若只部署静态前端：

- 页面可以打开
- 但 `/api/chat` 和 `/api/vision` 不会真正工作
- 在线模式就会失效，前端只能回退到离线模式

### 12.3 适合的部署方式

#### 方案 A：前后端同域部署

最省心：

- 前端静态资源和后端服务使用同一个域名
- 前端继续请求 `/api/chat`、`/api/vision`
- 由服务器或反向代理把 `/api/*` 转发给后端

#### 方案 B：前后端分开部署

- 前端部署在一个域名
- 后端单独部署在另一个域名
- 前端需要设置：

```env
VITE_API_BASE_URL=https://你的后端域名
```

#### 方案 C：Vercel 部署

如果已经把 `server/index.mjs` 拆成 `api/*.mjs` 形式的 Serverless Functions，那么可以部署为：

- 前端：Vite 静态页面
- 后端：Vercel Functions
- 环境变量：在 Vercel 项目后台配置

## 13. 在线模式与离线模式说明

这个项目特意保留了“在线不可用时仍可工作”的能力。

### 在线模式触发条件

- `/api/health` 返回 `ok: true`
- `chatConfigured: true`
- 当用户使用图片功能时，还需要 `visionConfigured: true`

### 离线模式触发场景

- 后端未启动
- 服务部署缺失 `/api/*`
- 环境变量没配好
- API Key 无效
- 上游接口返回错误

### 离线模式能做什么

- 识别常见科目名称
- 根据用户文字提取部分结构化信息
- 按规则生成任务
- 合并到本地课程/任务数据中

所以当你看到“已切换为离线模式”时，不代表项目完全坏了，而是进入了容错流程。

## 14. 常见问题排查

### 14.1 页面能打开，但提示“在线模型暂时不可用”

优先检查：

1. `DEEPSEEK_API_KEY` 是否正确
2. `/api/health` 返回里 `chatConfigured` 是否为 `true`
3. 上游接口是否返回 401 / 500
4. 部署平台环境变量修改后是否重新部署

### 14.2 `/api/health` 正常，但对话仍然失败

说明：

- 后端能访问
- 环境变量名可能没问题
- 但 Key 的值可能无效，或上游接口返回错误

这时应查看 `/api/chat` 的实际报错内容。

### 14.3 图片识别不可用

通常说明：

- `VISION_API_KEY` 未配置
- `/api/health` 中 `visionConfigured` 为 `false`
- 视觉接口地址或模型配置不匹配

### 14.4 本地改了 `.env` 但没有生效

修改 `.env` 后，需要重启后端。旧进程不会自动读取新值。

### 14.5 任务生成了，但看起来不像真实模型输出

说明系统可能已经回退到离线模式，此时任务是由前端规则引擎生成的。
